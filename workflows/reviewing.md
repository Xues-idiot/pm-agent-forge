# Reviewing Workflow

审核阶段，确保产出符合要求。

## 何时使用

- 功能开发完成后
- PRD 撰写完成后
- 活动设计完成后
- 上线前审核
- 重大决策前评审

## 审核类型

| 类型 | 时机 | 参与者 | 关注点 |
|------|------|--------|--------|
| 设计评审 | 开发前 | 产品、设计、技术 | 方案可行性 |
| 代码评审 | 开发中/后 | 开发、架构 | 代码质量 |
| 需求评审 | PRD 完成 | 产品、技术、测试 | 需求完整性 |
| 效果评审 | 上线后 | 产品、运营 | 目标达成 |
| 复盘评审 | 项目结束 | 全体 | 经验教训 |

## 评审流程

```
1. 评审准备 → 2. 评审执行 → 3. 问题跟踪 → 4. 改进验证
```

### 1. 评审准备

**评审前检查：**
```markdown
## 评审前检查

### 材料准备
- [ ] 评审材料已提前发送
- [ ] 材料内容完整
- [ ] 评审议程已通知

### 参与者确认
- [ ] 关键人员已确认参加
- [ ] 职责已明确
- [ ] 时间已约定

### 背景信息
- 相关文档：
- 历史决策：
- 约束条件：
```

**评审议程：**
```markdown
## 评审议程

### 基本信息
- 主题：
- 时间：
- 地点：
- 主持人：

### 议程
| 时间 | 内容 | 负责人 |
|------|------|--------|
| 5min | 开场 | 主持人 |
| 20min | 内容讲解 | 讲解人 |
| 30min | 讨论和提问 | 全体 |
| 5min | 总结和下一步 | 主持人 |
```

### 2. 评审执行

**两阶段评审（参考 Superpowers）：**

**阶段一：规格合规**
```markdown
## 阶段一：规格合规检查

### 需求理解
- [ ] 理解需求背景
- [ ] 理解用户故事
- [ ] 理解验收标准

### 规格覆盖
- [ ] 所有功能点已覆盖
- [ ] 边界情况已考虑
- [ ] 非功能需求已明确

### 疑问澄清
- [ ] 模糊点已澄清
- [ ] 假设已确认
- [ ] 遗漏已补充
```

**阶段二：质量检查**
```markdown
## 阶段二：质量检查

### 实现质量
- [ ] 技术方案可行
- [ ] 代码规范符合
- [ ] 安全性考虑

### 可维护性
- [ ] 架构合理
- [ ] 扩展性考虑
- [ ] 文档完整

### 测试覆盖
- [ ] 测试用例设计
- [ ] 风险场景覆盖
- [ ] 回归测试计划
```

## PRD 评审清单

评审 PRD 时检查：
- [ ] 目标清晰可衡量
- [ ] 用户故事完整
- [ ] 功能需求覆盖全面
- [ ] 验收标准 Given-When-Then
- [ ] 边界情况已考虑
- [ ] 非功能需求明确
- [ ] 优先级定义合理
- [ ] 技术方案可行

## 活动评审清单

评审活动方案时检查：
- [ ] 活动目标明确
- [ ] 目标用户清晰
- [ ] 活动机制简单易懂
- [ ] 奖励成本可控
- [ ] 触达计划覆盖目标用户
- [ ] 防刷机制到位
- [ ] 数据埋点完整

## 红牌警告

- 评审材料不完整
- 关键人员未参与
- P0 问题未修复就通过
- 问题修复未验证
- 评审流于形式
- 决策无记录
