# Planning Workflow

制定详细的实施计划。

## 何时使用

- 头脑风暴后确定方向
- 开始新功能开发
- 复杂任务需要规划
- 项目启动阶段

## 流程

```
1. 范围定义 → 2. 任务分解 → 3. 资源规划 → 4. 风险评估 → 5. 计划输出
```

### 1. 范围定义

**WBS（工作分解结构）：**
```markdown
## 项目范围

### 包含范围
- 功能1
- 功能2
- 功能3

### 排除范围
- 功能A（原因）
- 功能B（原因）

### 里程碑
| 阶段 | 交付物 | 日期 |
|------|--------|------|
| M1 | 需求确认 | 3/1 |
| M2 | 设计完成 | 3/10 |
| M3 | 开发完成 | 4/1 |
| M4 | 上线 | 4/15 |
```

```markdown
### Phase 2: 开发阶段
```


### 4. 风险评估

**风险评估矩阵：**

| 风险 | 可能性 | 影响 | 应对 |
|------|--------|------|------|
| 风险1 | 高/中/低 | 高/中/低 | 应对方案 |
| 风险2 | | | |

**风险应对策略：**

| 策略 | 适用场景 |
|------|---------|
| 规避 | 放弃高风险方案 |
| 转移 | 购买保险、外包 |
| 减轻 | 增加测试、备份 |
| 接受 | 低风险、可承受 |

## 计划检查清单

- [ ] 范围明确
- [ ] 任务分解到可执行粒度
- [ ] 有明确的里程碑
- [ ] 资源需求清晰
- [ ] 风险已识别有应对
- [ ] 依赖关系明确
- [ ] 时间线合理

## 红牌警告

- 计划过于乐观
- 忽略依赖关系
- 没有考虑风险
- 任务粒度太粗
- 资源估算不准确
- 缺少沟通计划
