# Agent Evolution Skill

让 Pi/Nu 能够持续学习和自我进化的技能。

## 何时使用

- Pi/Nu 需要添加新功能
- 用户反馈需要改进
- 定期自我检查和优化
- 发现更好的方法或模式

## 进化循环

```
1. 自我评估 → 2. 学习新知识 → 3. 制定改进计划 → 4. 实施改进 → 5. 验证效果 → 6. 记录进化
```

### 1. 自我评估

问自己：
- 当前最薄弱的地方是什么？
- 用户反馈最多的问题是什么？
- 最近学到的知识有没有应用？
- 哪些功能使用频率最低？

### 2. 学习新知识

从以下来源学习：
- Tavily 搜索相关主题
- GitHub trending 找灵感
- 现有代码库分析
- 用户反馈分析

### 3. 制定改进计划

创建具体、可执行的改进计划：
```markdown
## 改进计划

### 优先级 P0
1. [具体改进项]
2. [具体改进项]

### 优先级 P1
1. [具体改进项]
```

### 4. 实施改进

遵循最小化原则：
- 每次只做一个小改进
- 确保改进可验证
- 保持代码简洁

### 5. 验证效果

- 运行测试
- 检查是否解决了问题
- 观察是否有新问题引入

### 6. 记录进化

更新 PROGRESS.md：
```markdown
## 进化记录 - [日期]

### 本轮改进
- [改进项 1]
- [改进项 2]

### 学到的新知识
- [知识 1]
- [知识 2]

### 下一步计划
- [计划 1]
```

## 自我反思问题

每次进化时问自己：

| 问题 | 为什么重要 |
|------|----------|
| 这个改进对用户有什么价值？ | 确保改进有意义 |
| 会不会引入新的问题？ | 避免拆东墙补西墙 |
| 这个改进可持续吗？ | 避免临时方案 |
| 有没有更简单的方案？ | YAGNI 原则 |

## 红牌警告（停止信号）

- 改动太大，超过 200 行
- 涉及多个模块/功能
- 没有测试验证
- 用户没有要求

## 进化模式

### 渐进式进化
- 每次一个小改进
- 快速验证
- 持续迭代

### 突破式进化
- 积累多个小改进后
- 重大架构调整
- 需要更仔细的规划

## 集成 TDD（来自 Superpowers）

改进时遵循 TDD：
1. **RED**: 先写测试描述期望
2. **GREEN**: 写最小代码实现
3. **REFACTOR**: 清理代码

## 集成 Subagent（来自 Superpowers）

复杂改进可以用 Subagent：
- 派发 research agent 调研
- 派发 implementer agent 实现
- 派发 reviewer agent 审核

## 检查清单

进化完成后检查：
- [ ] PROGRESS.md 已更新
- [ ] 测试通过
- [ ] 没有引入新的警告
- [ ] 改进可验证
