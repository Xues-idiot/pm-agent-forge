# 服务网格 (Service Mesh)

## Overview

服务网格是管理微服务间通信的基础设施层，提供了服务发现、负载均衡、熔断限流、安全认证等能力。涵盖Istio/Linkerd等主流方案。

## 何时使用

- 微服务数量多，治理复杂
- 需要统一流量管理
- 需要mTLS安全通信
- 需要灰度发布能力
- 需要分布式追踪

## 流程

1. **方案选型**：选择Istio/Linkerd等方案
2. **架构设计**：设计网格架构
3. **流量管理**：配置流量规则
4. **安全配置**：配置mTLS和服务账号
5. **可观测性**：配置监控和追踪
6. **运维治理**：制定运维流程
7. **性能调优**：优化Sidecar资源

## 模板

### 服务网格架构模板
```
## 架构方案
- 产品: [Istio/Linkerd]
- 版本: [X]
- 部署模式: [Sidecar/Headless]

## 核心组件
| 组件 | 职责 | 配置 |
|------|------|------|
| Pilot | 流量管理 | - |
| Citadel | 安全认证 | mTLS模式 |
| Mixer | 策略和遥测 | - |

## 流量管理规则
- 负载均衡: [轮询/权重/地域]
- 超时重试: [超时Xms, 重试X次]
- 熔断策略: [错误率>X%触发]

## 安全配置
- mTLS模式: [Permissive/Strict/Disable]
- 认证策略: [PeerAuthentication]
- 授权策略: [AuthorizationPolicy]

## 可观测性
- 指标: [Prometheus+Grafana]
- 日志: [结构化日志]
- 追踪: [Jaeger/Zipkin]
```

## 检查清单

- [ ] 架构方案选型合理
- [ ] Sidecar资源配额合理
- [ ] mTLS配置正确
- [ ] 流量治理规则完善
- [ ] 监控告警已配置
- [ ] 有灰度发布能力

## 红牌警告

- Sidecar资源消耗过大
- mTLS配置错误导致服务不可用
- 缺乏流量治理导致故障蔓延
- 监控缺失无法定位问题
