# System Architecture Design Skill

设计可扩展、可维护的系统架构。

## 何时使用

- 新系统架构设计
- 技术选型决策
- 系统架构评审
- 微服务拆分设计
- 高并发架构规划

## 架构设计流程

```
1. 需求分析 → 2. 架构设计 → 3. 技术选型 → 4. 架构评审 → 5. 文档输出
```

### 1. 需求分析

**非功能需求：**

```markdown
## 非功能需求

### 性能指标
| 指标 | 要求 |
|------|------|
| QPS | >10000 |
| 延迟 | P99 < 200ms |
| 可用性 | >99.9% |

### 扩展性
| 维度 | 要求 |
|------|------|
| 用户规模 | 支持1亿用户 |
| 数据量 | 支持10TB数据 |
| 并发 | 支持10万并发 |

### 安全要求
- 数据加密
- 权限控制
- 审计日志
```

### 2. 架构设计

**常用架构模式：**

```markdown
## 架构模式选择

### 单体架构
- 适用：小型项目、快速迭代
- 优点：简单、部署方便
- 缺点：扩展性差

### 微服务架构
- 适用：大型项目、需要高扩展
- 优点：独立部署、技术灵活
- 缺点：复杂度高、运维成本大

### 事件驱动架构
- 适用：实时系统、异步处理
- 优点：解耦、扩展性好
- 缺点：一致性挑战

### CQRS架构
- 适用：读多写少场景
- 优点：读写分离、性能优化
- 缺点：复杂度高
```

**架构设计模板：**

```markdown
## 系统架构设计

### 整体架构图
```
[客户端]
    ↓
[网关层]
    ↓
[服务层] → [缓存层] → [存储层]
    ↓           ↓
[消息队列]   [搜索层]
```

### 模块划分
| 模块 | 职责 | 技术栈 |
|------|------|--------|
| 用户服务 | 用户管理 | Node.js |
| 订单服务 | 订单处理 | Python |
| 支付服务 | 支付集成 | Java |
```

### 3. 技术选型

**技术选型矩阵：**

```markdown
## 技术选型

### 语言/框架
| 场景 | 选项1 | 选项2 | 选择 |
|------|-------|-------|------|
| Web服务 | Node.js | Go | 按团队熟悉度 |
| 数据处理 | Python | Java | Python生态丰富 |
| 高并发 | Go | Rust | Go平衡性能和开发效率 |

### 数据库选型
| 数据类型 | 选项 | 说明 |
|---------|------|------|
| 关系数据 | PostgreSQL | 事务支持强 |
| 文档数据 | MongoDB | 灵活Schema |
| 缓存数据 | Redis | 高性能 |
| 搜索 | Elasticsearch | 全文搜索 |

### 中间件选型
| 组件 | 选项 | 说明 |
|------|------|------|
| 消息队列 | Kafka/RabbitMQ | 按吞吐量选 |
| 网关 | Kong/Nginx | 按功能需求 |
| 容器 | Docker/K8s | 按规模 |
```

### 4. 架构评审

**评审checklist：**

```markdown
## 架构评审

### 完整性
- [ ] 功能需求覆盖
- [ ] 非功能需求满足
- [ ] 边界清晰

### 可行性
- [ ] 技术可行性验证
- [ ] 团队能力匹配
- [ ] 成本可控

### 扩展性
- [ ] 水平扩展能力
- [ ] 模块解耦
- [ ] 数据分片

### 安全性
- [ ] 认证授权
- [ ] 数据加密
- [ ] 审计日志

### 可靠性
- [ ] 故障隔离
- [ ] 容灾备份
- [ ] 监控告警
```

- 过度设计
- 技术选型追新
- 忽略团队能力
- 扩展性不足
- 安全性缺失
- 单点故障风险
