# Product Planning

产品规划，定义产品愿景和制定产品路线图。

## 何时使用

- 产品愿景定义
- 产品路线图制定
- 产品规划周期
- 资源分配规划
- 版本规划

## 产品规划框架

### 产品愿景

```markdown
## 产品愿景

### 愿景定义
| 要素 | 内容 | 示例 |
|------|------|------|
| 愿景 | 5-10年目标 | 成为行业领先的XXX |
| 使命 | 存在的意义 | 让XX变得更容易 |
| 价值观 | 核心原则 | 用户第一、持续创新 |

### 愿景模板
```markdown
## 产品愿景

**愿景**：5-10年后我们要成为...

**使命**：我们存在的意义是...

**核心价值**：
- 用户至上
- 持续创新
- 追求卓越
```
```

## 产品路线图

### 规划周期

|------|------|------|

| 中期规划 | 6-12月 | 版本规划 |

## 产品路线图

### Q1

|------|------|----------|

### Q2
| 版本 | 主题 | 核心功能 |
|------|------|----------|
| V2.0 | 高级功能 | XX、XX |
```
```

## 需求收集

| 来源 | 类型 | 占比 |
|------|------|------|
| 用户反馈 | 定性 | 30% |

```markdown
## 需求池

| ID | 需求 | 来源 | 优先级 | 状态 |
|----|------|------|--------|------|

```
```

### 需求分类

```markdown
## 需求分类

### 按类型分
| 类型 | 说明 | 占比 |
|------|------|------|
| 新功能 | 全新功能 | 20% |
| 功能优化 | 体验提升 | 40% |
| 技术债务 | 重构优化 | 20% |
| Bug修复 | 问题修复 | 20% |

### 按价值分
| 象限 | 说明 | 策略 |
|------|------|------|
| 高价值+高紧急 | 核心需求 | 立即做 |
| 高价值+低紧急 | 重要需求 | 规划做 |
| 低价值+高紧急 | 紧急需求 | 尽快做 |
| 低价值+低紧急 | 可做可不做 | 尽量不做 |
```

## 版本规划

### 版本节奏

```markdown
## 版本节奏

### 发布周期
| 模式 | 周期 | 适用 | 说明 |
|------|------|------|------|
| 敏捷迭代 | 2周 | 互联网产品 | 快速试错 |
| 月度发布 | 1月 | SaaS产品 | 稳定为主 |
| 季度发布 | 3月 | 企业软件 | 大版本 |
| 大版本+小版本 | 不固定 | 平台产品 | 重大升级 |

### 版本命名
```markdown
## 版本命名

- 主版本：功能重大更新
- 次版本：功能新增优化
- 补丁版本：问题修复

示例：
- V1.0.0（首发）
- V1.1.0（次版本更新）
- V1.1.1（补丁修复）
```
```

### 版本规划流程

```markdown
## 版本规划

### 规划流程
1. 需求评审：确定版本范围
2. 技术评估：评估实现难度
3. 工作量估算：评估开发周期
4. 排期确认：确认发布时间
5. 开发执行：迭代开发
6. 测试验收：质量把控
7. 发布上线：版本发布

### 规划会议
```markdown
## 版本规划会议

### 参会人
- 产品经理
- 技术负责人
- 设计
- 测试

### 议程
1. 上期回顾（10min）
2. 需求评审（30min）
3. 排期确认（20min）
4. 风险识别（10min）
```
```

## 检查清单

- [ ] 产品愿景清晰
- [ ] 路线图规划完整
- [ ] 需求收集系统化
- [ ] 版本节奏合理
- [ ] 资源配置恰当
- [ ] 风险预案准备

## 红牌警告

- 愿景假大空
- 路线图过于详细或过于模糊
- 需求没有优先级
- 版本延期常态化
- 资源浪费或不足

## 好/坏对比

**好**：
```
产品规划：
- 愿景清晰且可实现
- 路线图有层次感
- 需求优先级明确
- 版本节奏稳定
→ 团队有方向、执行有力
```

**坏**：
```
产品规划：
- 愿景年年变
- 路线图是鬼话
- 需求没有优先级
- 版本经常延期
→ 团队迷茫、执行混乱
```
