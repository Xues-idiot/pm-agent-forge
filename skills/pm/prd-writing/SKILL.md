# PM PRD Writing Skill

写好 PRD（产品需求文档）的专业技能。

## 核心原则

| 原则 | 说明 |
|------|------|
| **清晰** | 每个需求只有一种解释 |
| **完整** | 覆盖所有场景，包括边界情况 |
| **一致** | 文档内部不矛盾 |
| **可验证** | 每个需求都可以测试 |

## PRD 结构（参考 MiroFish 图谱构建）

```markdown
# PRD: [产品名] - [功能名]

## 1. 概述
- 背景
- 目标
- 成功指标

## 2. 用户分析
- 目标用户
- 用户故事
- 用户旅程

## 3. 功能需求
- 核心功能
- 扩展功能
- 边界情况

## 4. 非功能需求
- 性能
- 安全
- 兼容性

## 5. 技术方案
- 系统架构
- 数据模型
- API 设计

## 6. 测试计划
- 验收标准
- 测试用例

## 7. 排期
- 里程碑
- 风险评估
```

## PRD 写作技巧

### 用户故事格式

```markdown
作为 [用户类型]
我想要 [功能]
以便 [价值]
```

**好例子：**
```
作为 产品经理
我想要 查看用户留存漏斗
以便 发现用户流失节点
```

**坏例子：**
```
作为用户，我想要一个好的功能
```

### 验收标准格式

```markdown
**Given** [前置条件]
**When** [操作]
**Then** [预期结果]
```

**好例子：**
```
Given 用户已登录
When 用户点击"导出报告"
Then 下载包含所有数据的 CSV 文件
```

### 优先级定义

| 优先级 | 说明 | 例子 |
|--------|------|------|
| P0 | 必须有，核心功能 | 登录、核心业务流程 |
| P1 | 应该有，重要功能 | 通知、导出 |
| P2 | 可以有，辅助功能 | 主题定制 |
| P3 | 最好有，体验优化 | 动画、微交互 |

## PRD 自检清单

写完后检查：

- [ ] 每个功能有清晰的目标
- [ ] 每个用户故事可验证
- [ ] 没有模糊的词（"可能"、"也许"、"用户友好"）
- [ ] 边界情况已覆盖
- [ ] 技术方案可行
- [ ] 优先级合理
- [ ] 有测试计划

## PRD 审查问题

评审时问：

1. 用户真的需要这个功能吗？
2. 这个功能能解决什么问题？
3. 怎样算成功？
4. 有没有更简单的方案？
5. 风险是什么？

## 与其他技能集成

### 集成 User Research
写 PRD 前先做用户调研，确认需求真实存在。

### 集成 Competitive Analysis
分析竞品是否有类似功能，如何做得更好。

### 集成 Agent Evolution
PRD 完成后，检查是否需要更新 Agent 能力。

## 反模式（避免）

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 功能堆砌 | 没有重点 | 聚焦核心价值 |
| 模糊描述 | 无法验证 | 具体可测试 |
| 技术细节过多 | 限制了实现自由度 | 关注"是什么"而非"怎么做" |
| 忽略边界情况 | 上线后 Bug 多 | 列出所有场景 |
| 没有优先级 | 开发无焦点 | 清晰 P0/P1/P2 |
