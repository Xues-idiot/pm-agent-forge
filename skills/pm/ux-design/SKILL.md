# UX Design

用户体验设计，UX原则、设计评审和可用性测试。

## 何时使用

- 设计用户体验流程
- 进行设计评审
- 开展可用性测试
- 制定UX规范
- 优化用户旅程

## UX设计原则

### Nielsen 10可用性原则

| 原则 | 说明 | 检查项 |
|------|------|--------|
| 系统状态可见 | 用户知道在哪、在做什么 | 有进度指示？ |
| 系统与现实匹配 | 用用户语言 | 图标易懂？ |
| 用户控制 | 用户可以撤销操作 | 有返回按钮？ |
| 一致性 | 平台内一致 | 同类操作方式相同？ |
| 防错 | 防止错误发生 | 有确认提示？ |
| 识别而非回忆 | 减少记忆负担 | 选项可见？ |
| 灵活高效 | 满足新手和老手 | 有快捷方式？ |
| 美观简洁 | 去除冗余信息 | 界面简洁？ |
| 容错 | 帮助从错误恢复 | 错误信息有用？ |
| 帮助文档 | 需要时提供帮助 | 有使用指引？ |

### UX设计流程

```
用户研究 → 需求分析 → 信息架构 → 交互设计 → 视觉设计 → 原型测试
```

## 信息架构

### 导航设计原则

```markdown
## 导航设计

### 导航类型
| 类型 | 适用场景 | 示例 |
|------|----------|------|
| 顶部导航 | 内容类网站 | 新闻、博客 |
| 侧边导航 | 功能型应用 | SaaS、管理后台 |
| 底部导航 | 移动端 | APP |
| 面包屑 | 深层结构 | 电商、分类信息 |

### 导航原则
✓ 位置一致
✓ 标签清晰
✓ 分组合理
✓ 有搜索能力
```

### 卡片分类

```markdown
## 卡片分类

### 目的
让用户将功能分组，找出用户的mental model

### 方法
1. 准备卡片：每个功能一张卡片
2. 用户分组：让用户自己分
3. 命名：为每组命名
4. 分析：找出共同模式和边界

### 分析模板
| 卡片 | 用户A | 用户B | 用户C | 结论 |
|------|-------|-------|-------|------|
| 功能1 | 组A | 组A | 组B | 争议 |
| 功能2 | 组B | 组B | 组B | 一致 |
```

## 交互设计

### 交互设计模式

| 模式 | 适用场景 | 示例 |
|------|----------|------|
| 渐进披露 | 复杂操作 | 分步骤表单 |
| 实时搜索 | 大量数据 | 输入即搜索 |
| 拖拽排序 | 排列顺序 | 列表排序 |
| 模态对话框 | 快速任务 | 确认操作 |
| 侧滑面板 | 次要操作 | 设置面板 |

### 反馈设计

```markdown
## 反馈设计

### 反馈类型
| 类型 | 时机 | 示例 |
|------|------|------|
| 立即反馈 | 操作后<1秒 | 点击变色 |
| 过程反馈 | 操作进行中 | 进度条 |
| 结果反馈 | 操作完成 | 成功/失败提示 |

### 反馈强度
- 弱反馈：轻微动效、颜色变化
- 中反馈：提示消息
- 强反馈：模态弹窗、声音
```

## 可用性测试

### 测试类型

| 类型 | 样本量 | 时间 | 适用场景 |
|------|--------|------|----------|
| 快速测试 | 5人 | 30分钟 | 发现大问题 |
| 正式测试 | 8-10人 | 1-2小时 | 完整评估 |
| 田野测试 | 5-8人 | 半天 | 真实环境 |

### 测试任务模板


## 设计评审

### 设计评审检查清单

```markdown
## 设计评审

### 功能完整性
- [ ] 需求功能都有覆盖？
- [ ] 边界情况有处理？
- [ ] 错误状态有设计？

### 可用性
- [ ] 操作路径合理？
- [ ] 信息层次清晰？
- [ ] 帮助信息足够？

### 一致性
- [ ] 与现有设计一致？
- [ ] 平台规范一致？
- [ ] 术语一致？

### 可实现性
- [ ] 技术可行？
- [ ] 性能可接受？
- [ ] 适配主流设备？
```

## 检查清单

- [ ] 用户旅程清晰
- [ ] 信息架构合理
- [ ] 交互模式适当
- [ ] 反馈机制完善
- [ ] 可用性测试通过

## 红牌警告

- 没有用户研究就设计
- 不做可用性测试就上线
- 设计与实现脱节
- 忽视边缘用户

## 好/坏对比

**好**：
```
设计前：用户访谈，了解用户mental model
设计中：可用性测试，及时发现问题
上线后：数据分析，持续优化
```

**坏**：
```
"设计师说了算"
"开发说实现不了就算了"
"没时间测试直接上线"
```
