# Product Lifecycle

产品生命周期管理，从概念到发布再到迭代。

## 何时使用

- 新产品开发
- MVP设计
- 产品发布
- 版本迭代
- 产品优化

## 概念阶段

### 需求发现

```markdown
## 需求发现

### 需求来源
| 来源 | 说明 | 占比 |
|------|------|------|
| 用户反馈 | 客服、调研 | 30% |
| 市场洞察 | 竞品、行业 | 25% |
| 数据分析 | 行为数据 | 20% |
| 业务战略 | 公司目标 | 15% |
| 技术驱动 | 新技术 | 10% |

## 需求评估

## MVP阶段

### MVP设计

```markdown
## MVP设计

### MVP原则
| 原则 | 说明 |
|------|------|
| 最小功能 | 只做核心功能 |
| 快速验证 | 尽快上线验证 |
| 学习导向 | 验证假设而非完美 |

### MVP类型
| 类型 | 适用 | 示例 |
|------|------|------|
| 纸面原型 | 概念验证 | 演示 |
| 视频原型 | 体验验证 | Demo |
| 众筹MVP | 市场验证 | 小规模售卖 |
| 绿野仙踪 | 功能验证 | 人工后台 |

### MVP功能清单
```markdown
## MVP功能

### 核心功能（P0）
- 功能A
- 功能B
- 功能C

### 重要功能（P1）
- 功能D
- 功能E

### nice to have（P2）
- 功能F
- 功能G
```
```

## 开发阶段

### 产品规划

```markdown
## 产品规划

### 迭代规划
| 版本 | 周期 | 目标 |
|------|------|------|
| V1.0 | 6-8周 | MVP发布 |
| V1.1 | 2周 | 核心反馈优化 |
| V2.0 | 4周 | 重要功能 |
| V2.1 | 2周 | 体验优化 |

### 开发流程
```markdown
## 开发流程

1. 需求评审 → 2. 设计 → 3. 开发 → 4. 测试 → 5. 发布

### 各阶段输出
- 需求：PRD
- 设计：原型、UI
- 开发：代码
- 测试：测试用例、报告
- 发布：上线方案
```
```

```markdown

### 发布流程
1. 预发布验证
2. 正式发布
3. 监控观察
4. 效果验证

### 监控指标
| 指标 | 关注点 |
|------|--------|
| 错误率 | 有无上升 |
| 响应时间 | 是否正常 |
| 核心功能 | 是否可用 |
| 业务数据 | 是否正常 |
```

## 迭代阶段

### 迭代管理

```markdown
## 迭代管理

### 迭代周期
| 类型 | 时长 | 适用 |
|------|------|------|
| 敏捷迭代 | 1-2周 | 互联网 |
| 月度迭代 | 1月 | SaaS |
| 季度迭代 | 3月 | 企业软件 |

### 迭代流程
```markdown
## 迭代流程

### 两周迭代
- Day 1-2: 需求评审、规划
- Day 3-10: 设计+开发
- Day 11: 测试
- Day 12: 发布
- Day 13-14: 复盘、规划
```
```

### 数据驱动迭代

```markdown
## 数据驱动

### 监控指标
| 指标 | 定义 | 目标 |
|------|------|------|
| 功能使用率 | 使用人数/总人数 | >30% |
| 功能满意度 | 评分 | >4分 |
| 转化率 | 完成目标/访问 | >10% |
| 错误率 | 错误次数 | <1% |

## 检查清单

- [ ] 需求评估到位
- [ ] MVP设计合理
- [ ] 开发流程规范
- [ ] 发布准备充分
- [ ] 迭代管理科学
- [ ] 数据驱动优化

## 红牌警告

- 需求不清晰就开发
- MVP过于复杂
- 测试不充分
- 发布无回滚方案
- 不看数据
- 迭代无重点

## 好/坏对比

**好**：
```
产品生命周期：
- 需求验证充分
- MVP快速验证
- 开发流程规范
- 发布准备到位
- 数据驱动迭代
→ 快速验证、持续优化
```

**坏**：
```
产品生命周期：
- 需求拍脑袋
- MVP过于复杂
- 测试走过场
- 发布无准备
- 不看数据
→ 开发周期长、失败率高
```
