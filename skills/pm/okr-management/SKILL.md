# OKR Management

OKR目标管理，设定目标、追踪进展、对齐团队。

## 何时使用

- 制定季度OKR
- 对齐团队目标
- 追踪OKR进展
- 复盘OKR结果
- 推动目标落地

## OKR框架

### OKR vs KPI

```markdown
## OKR vs KPI

| 维度 | OKR | KPI |
|------|-----|-----|
| 目的 | 驱动突破 | 维持标准 |
| 目标 | 定性/定量 | 定量 |
| 周期 | 通常季度 | 通常月度 |
| 评分 | 0-1.0 | 百分比 |
| 导向 | 野心/挑战 | 达成/稳定 |

### 选择指南
- 需要突破：OKR
- 保持稳定：KPI
- 组合使用：OKR定方向，KPI保执行
```

### OKR制定流程

```
战略对齐 → 目标设定 → 关键结果 → 对齐沟通 → 追踪复盘
```

## 目标(Objective)撰写

### 好的目标特征

```markdown
## 好的目标特征

### SMART-O
| 特征 | 说明 | 示例 |
|------|------|------|
| Specific | 具体明确 | "提升用户留存" |
| Motivational | 有激励性 | 让团队兴奋 |
| Aligned | 对齐战略 | 支持公司目标 |
| Responsible | 有责任感 | 有人负责 |
| Time-bound | 有时间 | Q2完成 |

### 好的目标示例
✓ "让用户爱上我们的产品"
✓ "成为设计师首选的工具"
✓ "打造最高效的团队协作体验"

### 坏的目标示例
✗ "提升指标"
✗ "做好用户体验"
✗ "增加收入"
```

### 目标撰写模板

```markdown
## 目标撰写

### 模板
[动词] + [做什么] + [给谁/解决什么问题]

### 示例
| 层级 | 目标 |
|------|------|
| 公司 | 让更多企业采用我们的协作工具 |
| 团队 | 打造让设计师爱不释手的产品体验 |
| 个人 | 帮助用户高效完成日常工作 |
```

## 关键结果(Key Results)

### KR类型

```markdown
## KR类型

### 定量KR
| 类型 | 示例 |
|------|------|
| 绝对值 | MAU达到100万 |
| 增长值 | 新增50万用户 |
| 百分比 | 付费转化率提升至5% |
| 相对值 | NPS提升20分 |

### 里程碑KR
| 类型 | 示例 |
|------|------|
| 交付型 | 上线新功能V2 |
| 探索型 | 完成3个用户研究 |
| 改进型 | 延迟降低50% |
```

### KR撰写原则

```markdown
## KR撰写原则

### 好的KR
✓ 可衡量：有明确数值
✓ 有挑战：达成0.7需要努力
✓ 具体清晰：知道做什么
✓ 有时间：Q2完成

### KR格式
"KR[数字]：[是什么] [衡量指标] [目标值]"

### 示例
KR1：季度收入达到¥1000万（当前¥700万）
KR2：付费转化率从3%提升至5%
KR3：Q2上线3个核心功能
KR4：用户满意度NPS达到50+
```

### KR自检清单

```markdown
## KR自检

### 评估KR质量
- [ ] KR可量化？
- [ ] 有挑战性（0.7分需要努力）？
- [ ] 有期限（季度末完成）？
- [ ] 多个KR能支撑Objective？

### 量化检查
| 问题 | 好的KR | 坏的KR |
|------|--------|--------|
| 完成标准清晰？ | ✅ MAU≥100万 | ❌ 用户数增长 |
| 如何衡量？ | ✅ Excel可算 | ❌ 体验提升 |
| 有挑战？ | ✅ 需努力达成 | ❌ 轻松可达 |
```

## 对齐与落地

### 上下对齐

```markdown
## 对齐流程

### 自上而下
```
公司OKR
    ↓ 对齐
团队OKR
    ↓ 对齐
个人OKR
```

### 对齐检查
| 对齐关系 | 检查点 |
|----------|--------|
| 公司→团队 | 团队OKR支持公司OKR？ |
| 团队→个人 | 个人OKR支持团队OKR？ |
| 横向协作 | 相关团队知道彼此OKR？ |

### 协作KR
```markdown
## 协作KR

### 定义
需要多个团队/角色共同完成的KR

### 示例
| KR | 负责人 | 依赖方 |
|----|--------|--------|
| 上线新功能X | 产品+研发 | 设计提供UI |
| 提升NPS 20分 | 客服+产品 | 研发修复问题 |
```

## 追踪与复盘

### 追踪机制

```markdown
## OKR追踪

### 周检查
- 进展更新：KR进展%
- 阻塞问题：是否有？
- 调整需要：是否需要修改？

### 评分标准
| 分数 | 定义 | 说明 |
|------|------|------|
| 1.0 | 完美达成 | 不可能 |
| 0.7-0.9 | 达成 | 良好 |
| 0.4-0.6 | 部分达成 | 需改进 |
| 0-0.3 | 未达成 | 失败 |

### 评分原则
- 0.7 = 成功
- 0.7不是0.7是满分，是成功
- 长期0.9+ = 目标不够有挑战
```

## 检查清单

- [ ] 目标有激励性
- [ ] 目标对齐战略
- [ ] KR可量化有挑战
- [ ] 上下左右对齐
- [ ] 有追踪机制
- [ ] 有定期复盘

## 红牌警告

- OKR变成KPI（必须达成）
- 目标没有挑战性
- KR不可衡量
- 制定后不追踪
- OKR与绩效挂钩

## 好/坏对比

**好**：
```
OKR制定：
- 目标有挑战：MAU从100万→300万
- KR可衡量：Q2末达成
- 团队对齐：每个人知道贡献
→ 有动力、有方向
```

**坏**：
```
OKR制定：
- 目标保守：MAU从100万→105万
- KR写成KPI：MAU必须≥105万
- 不追踪不问进度
→ OKR变成KPI
```
