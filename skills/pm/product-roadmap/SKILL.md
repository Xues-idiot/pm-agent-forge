# Product Roadmap

产品路线图规划。

## 何时使用

- 制定产品路线图
- 规划长期产品方向
- 季度/年度规划
- 路线图沟通
- 优先级排序

## 路线图基础

### 路线图类型

```markdown
## 路线图类型

### 按时间分类
| 类型 | 时间跨度 | 特点 |
|------|----------|------|
| 战略路线图 | 3-5年 | 愿景/方向 |
| 年度路线图 | 1年 | 目标/里程碑 |
| 季度路线图 | 3个月 | 具体计划 |
| 迭代路线图 | 2-4周 | Sprint规划 |

### 按形式分类
| 类型 | 特点 | 适用 |
|------|------|------|
| 主题路线图 | 按主题分组 | 战略沟通 |
| 特性路线图 | 具体功能 | 开发团队 |
| 目标路线图 | 按OKR对齐 | 高层沟通 |
| 进度路线图 | 带状态进度 | 项目管理 |
```

## 规划流程

### 规划输入

```markdown
## 规划输入

### 输入维度
| 输入 | 内容 | 来源 |
|------|------|------|
| 战略 | 公司战略 | 领导层 |
| 用户 | 用户需求 | 用研/客服 |
| 竞品 | 竞品动态 | 竞分 |
| 数据 | 数据洞察 | 分析 |
| 技术 | 技术债务 | 研发 |
| 资源 | 团队能力 | 现状 |

### 环境分析
```markdown
## 环境分析

### SWOT分析
| 维度 | 内容 |
|------|------|
| S优势 | 产品优势 |
| W劣势 | 产品不足 |
| O机会 | 市场机会 |
| T威胁 | 竞争威胁 |

### 市场分析
- 市场规模
- 增长趋势
- 竞争格局
- 用户需求
```
```

### 规划框架

```markdown
## 规划框架

### MoSCoW优先级
| 类别 | 说明 | 占比 |
|------|------|------|
| Must | 必须有 | 60% |
| Should | 应该有 | 20% |
| Could | 可以有 | 15% |
| Won't | 这次不会有 | 5% |

### RICE评分
```markdown
## RICE评分

### 公式
RICE = (Reach × Impact × Confidence) / Effort

### 因素说明
- Reach：影响人数
- Impact：影响程度
- Confidence：确定性
- Effort：工作量
```
```

## 路线图设计

### 结构设计

```markdown
## 路线图结构

### 纵向结构
| 结构 | 内容 | 沟通对象 |
|------|------|----------|
| 按产品线 | 各产品线 | 高层 |
| 按用户群体 | 各用户群 | 产品/市场 |
| 按主题 | 主题/ Epic | 团队 |
| 按功能模块 | 功能模块 | 开发 |

### 横向结构
| 结构 | 时间 | 粒度 |
|------|------|------|
| 年度 | 12个月 | 季度 |
| 半年 | 6个月 | 月度 |
| 季度 | 3个月 | 周 |
| 月度 | 4周 | Sprint |
```

### 里程碑设计

```markdown
## 里程碑设计

### 里程碑类型
| 类型 | 说明 | 示例 |
|------|------|------|
| 战略里程碑 | 大方向 | 新市场进入 |
| 产品里程碑 | 产品发布 | V2.0发布 |
| 技术里程碑 | 技术突破 | 架构升级 |
| 业务里程碑 | 业务目标 | GMV过亿 |

### 里程碑原则
- 有明确交付物
- 有具体日期
- 可验证
- 有影响
```

## 路线图沟通

### 沟通对象

```markdown
## 路线图沟通

### 沟通矩阵
| 对象 | 关注点 | 形式 | 频率 |
|------|--------|------|------|
| 高层 | 战略/资源 | 1v1/会议 | 月度 |
| 研发 | 具体实现 | 会议 | 迭代 |
| 市场 | 对外承诺 | 预览版 | 季度 |
| 客服 | 客户预期 | 简版 | 按需 |

### 沟通原则
```markdown
## 沟通原则

### 高层沟通
- 关注战略价值
- 简化技术细节
- 提供选项

### 团队沟通
- 足够细节
- 解释为什么
- 接受反馈
```
```

## 路线图管理

### 评审流程

```markdown
## 路线图评审

### 评审节点
| 节点 | 时机 | 参与者 |
|------|------|--------|
| 战略评审 | 年度 | CEO/高层 |
| 产品评审 | 季度 | 产品+研发 |
| 迭代评审 | Sprint | 团队 |

### 评审要点
- 战略一致性
- 资源可行性
- 技术风险
- 依赖关系
- 优先级合理性
```

### 变更管理

```markdown
## 路线图变更

### 变更触发
| 触发 | 说明 | 处理 |
|------|------|------|
| 紧急需求 | 突发事件 | 快速插队 |
| 市场变化 | 竞品/用户 | 重新评估 |
| 资源变化 | 人力/预算 | 调整范围 |
| 技术风险 | 难度变大 | 延期/拆分 |

### 变更原则
- 小步快跑
- 及时沟通
- 记录变更
- 分析影响
```

## 红牌警告

- 没有用户洞察
- 不做优先级排序
- 承诺不兑现
- 不沟通变化
- 路线图过于详细
- 路线图过于粗略
- 不做复盘
