# Project Management

跨部门项目管理，协调团队、处理时间线和风险。

## 何时使用

- 跨部门协作项目
- 项目排期和里程碑
- 项目风险管理
- 团队协调
- 项目状态跟踪

## 项目管理流程

```
启动 → 规划 → 执行 → 监控 → 收尾
```

### 1. 项目启动

**项目章程模板**：
```markdown
## 项目章程

### 项目名称
[名称]

### 背景与目标
- 背景：[为什么做这个项目]
- 目标：[具体可衡量的目标]

### 成功标准
- [标准1]
- [标准2]

### 关键干系人
| 干系人 | 角色 | 关注点 |
|--------|------|--------|
| 研发负责人 | 执行 | 技术实现 |
| 设计负责人 | 执行 | UI/UX |
| 产品负责人 | 决策 | 业务价值 |

### 时间和资源
- 开始日期：
- 结束日期：
- 预算/人天：
```

### 2. 项目规划

**WBS分解**：
```markdown
## 工作分解结构(WBS)

### 一级模块
1. [模块1]
   1.1 [任务]
   1.2 [任务]
2. [模块2]
   2.1 [任务]
   2.2 [任务]
```

**里程碑计划**：
| 里程碑 | 完成日期 | 交付物 | 负责人 |
|--------|----------|--------|--------|
| M1: 需求确认 | 3/1 | PRD | PM |
| M2: 设计完成 | 3/15 | 设计稿 | UI |
| M3: 开发完成 | 4/15 | 代码 | 研发 |
| M4: 测试完成 | 4/30 | 测试报告 | QA |
| M5: 上线 | 5/1 | 生产环境 | DevOps |

### 3. 项目执行

**周报模板**：
```markdown
## 周报 [日期]

### 本周进度
| 任务 | 计划 | 实际 | 状态 |
|------|------|------|------|
| 任务1 | 100% | 100% | ✅ |
| 任务2 | 80% | 75% | ⚠️ |

### 风险与问题
| 风险/问题 | 影响 | 应对 |
|-----------|------|------|
| 风险1 | 高 | 措施 |

### 下周计划
- [ ] 任务1
- [ ] 任务2
```

### 4. 风险管理

**风险登记册**：
```markdown
## 风险登记册

| ID | 风险描述 | 概率 | 影响 | 应对策略 | 状态 |
|----|----------|------|------|----------|------|
| R1 | 人员离职 | 高 | 高 | 交叉培训 | 监控中 |
| R2 | 技术方案变更 | 中 | 中 | 预留buffer | 规避 |

### 风险应对策略
| 策略 | 适用场景 | 示例 |
|------|----------|------|
| 规避 | 风险高且无法控制 | 更换供应商 |
| 转移 | 风险高但可保险 | 外包非核心模块 |
| 缓解 | 风险可控 | 增加测试轮次 |
| 接受 | 风险低影响小 | 预留应急金 |
```

## 跨部门协作

### 协作常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 研发说"做不了" | 需求不合理或技术限制 | 拆解需求，共同评估 |
| 设计延期 | 需求变更频繁 | 需求冻结，变更流程 |
| 测试时间不足 | 计划太紧 | 预留buffer，分批迭代 |
| 优先级冲突 | 多项目并行 | 项目组合管理 |

### 沟通机制

```markdown
## 沟通机制

### 日常
- 站会：每日15分钟，阻塞问题
- 周会：每周项目状态同步

### 阶段评审
- 需求评审：研发、设计、PM
- 设计评审：PM、设计、研发
- 测试评审：QA、研发、PM

### 决策升级
- 阻塞问题 → 立即升级
- 范围变更 → 24小时内决策
- 重大风险 → 48小时内解决
```

## 红牌警告

- 没有项目章程就开始开发
- 需求没有冻结就开发
- 没有预留buffer的时间表
- 风险识别后不跟进

## 好/坏对比

**好**：
```
项目启动前：
- 制定项目章程
- 识别所有干系人
- 分解WBS
- 识别风险
- 制定沟通机制
→ 项目可控
```

**坏**：
```
"先做起来再说"
→ 范围蔓延
→ 工期延误
→ 质量下降
```
