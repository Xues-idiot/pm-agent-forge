# Four Risks Assessment

用Marty Cagan的Four Risks框架评估功能/项目风险。

## 何时使用

- 评估是否要做某个功能
- Sprint规划前的风险评估
- PRD评审
- 技术方案评审
- 项目启动前的自查

## 四种风险

| 风险 | 问题 | 验证方式 |
|------|------|----------|
| **Value Risk** | 用户会买/用这个吗？ | 用户访谈、支付意愿 |
| **Usability Risk** | 用户能搞懂怎么用吗？ | 可用性测试 |
| **Feasibility Risk** | 我们能造出来吗？ | 技术评估 |
| **Business Viability Risk** | 对业务有价值吗？ | 商业模型验证 |

## 详细风险框架

### Value Risk (价值风险)

**核心问题**: 用户会真正使用/购买这个功能吗？

**评估维度**:

| 维度 | 问题 |
|------|------|
| 需求真实性 | 用户真的说要这个，还是你以为他们要？ |
| 支付意愿 | 用户愿意付钱吗？ |
| 使用频率 | 用户会经常用还是用一次就不用了？ |
| 差异化 | 比现有方案好多少？ |

**去风险行动**:
- 10+用户访谈验证需求
- A/B测试或原型验证
- 分析竞品数据
- 用户反馈分析

### Usability Risk (可用性风险)

**核心问题**: 用户能轻松学会使用吗？

**评估维度**:

| 维度 | 问题 |
|------|------|
| 学习曲线 | 用户需要多久才能上手？ |
| 界面复杂 | UI是否直观？ |
| 认知负担 | 用户需要理解多少概念？ |
| 错误处理 | 用错了怎么办？ |

**去风险行动**:
- 可用性测试（5个用户）
- 创建原型
- 简化UI设计
- 添加引导/教程

### Feasibility Risk (可行性风险)

**核心问题**: 我们有技术能力实现吗？

**评估维度**:

| 维度 | 问题 |
|------|------|
| 技术能力 | 我们懂这个技术吗？ |
| 时间资源 | 能在时间内完成吗？ |
| 依赖 | 有外部依赖吗？ |
| 规模 | 能扩展到预期规模吗？ |

**去风险行动**:
- 技术原型验证
- 详细技术评估
- 识别关键技术风险
- 制定应急方案

### Business Viability Risk (商业可行性风险)

**核心问题**: 这个对业务真的有价值吗？

**评估维度**:

| 维度 | 问题 |
|------|------|
| 战略一致 | 符合公司战略吗？ |
| 商业模式 | 能变现吗？ |
| 成本 | 开发和维护成本合理吗？ |
| 竞争优势 | 给我们优势吗？ |

**去风险行动**:
- 商业模型分析
- 战略对齐检查
- 成本收益分析
- 竞争分析

## 检查清单

### 评估前
- [ ] 了解功能背景和目的
- [ ] 查看相关PRD/用户研究
- [ ] 识别关键假设

### 评估中
- [ ] Value Risk有用户证据
- [ ] Usability Risk有原型/测试
- [ ] Feasibility Risk有技术评估
- [ ] Business Risk有商业分析

### 评估后
- [ ] 每种风险有去风险行动
- [ ] 行动有明确负责人
- [ ] 有清晰的go/no-go决定

## 红牌警告

- 没有用户访谈就声称Value Risk低
- 技术方案没验证就说Feasibility Risk低
- 忽视Business Viability Risk
- 多个高风险一起出现

## 好/坏对比

**好**：
```
新功能AI推荐
- Value: 🔴高风险 → 只有3个用户访谈，需要更多
- Usability: 🟢低风险 → 有原型可用性测试
- Feasibility: 🟡中风险 → 需要验证AI模型
- Business: 🟡中风险 → 需要商业模型验证

结论：先做更多用户研究和技术验证
```

**坏**：
```
"这个功能很简单，没问题"
（没有实际评估任何风险）
```
