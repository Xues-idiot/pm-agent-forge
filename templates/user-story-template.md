# User Story 模板

> 用户故事标准格式

---

## 标准格式

```markdown
作为 [用户类型]
我想要 [功能]
以便 [价值/收益]
```

---

## 用户故事示例

### 示例 1：核心功能

```markdown
作为 产品经理
我想要 查看用户留存漏斗
以便 发现用户流失节点
```

### 示例 2：效率提升

```markdown
作为 运营人员
我想要 一键导出日报
以便 节省手动统计时间
```

### 示例 3：问题解决

```markdown
作为 客服
我想要 查看用户历史订单
以便 快速解答用户咨询
```

## 用户故事卡片模板

```markdown
## 用户故事卡片

### 故事描述
作为 [用户类型]
我想要 [功能]
以便 [价值]

### 验收标准（AC）

```markdown
Given [前置条件]
When [操作]
Then [预期结果]
```

### 边界条件
- [ ] 条件1
- [ ] 条件2

### 注释
[补充说明]
```

## 用户故事INVEST 原则

| 字母 | 含义 | 说明 |
|------|------|------|
| **I**ndependent | 独立性 | 故事间无依赖，可独立实现 |
| **N**egotiable | 可协商 | 不是固定合同，可讨论调整 |
| **V**aluable | 有价值 | 对用户或业务有价值 |
| **E**stimable | 可估算 | 团队能估算工作量 |
| **S**mall | 小 | 在一个迭代内完成 |
| **T**estable | 可测试 | 有明确的验收标准 |

## 故事点估算

### 斐波那契数列

| 故事点 | 复杂度 | 时间参考 |
|--------|--------|---------|
| 1 | 非常简单 | 0.5天 |
| 2 | 简单 | 1天 |
| 3 | 中等 | 1.5天 |
| 5 | 复杂 | 2-3天 |
| 8 | 很复杂 | 4-5天 |
| 13 | 巨大 | >5天 |

### 估算方法

**计划扑克：**
- 团队成员各自估算
- 最高和最低解释
- 重新估算
- 达成共识

## 用户故事优先级

| 优先级 | 定义 | 策略 |
|--------|------|------|
| **P0** | 必须有 | 核心流程 |
| **P1** | 应该有 | 重要但可延后 |
| **P2** | 可以有 | 锦上添花 |
| **P3** | 最好有 | 低优先级 |

### MoSCoW 方法

| 类别 | 含义 | 占比 |
|------|------|------|
| **M**ust | 必须有 | 60% |
| **S**hould | 应该有 | 20% |
| **C**ould | 可以有 | 15% |
| **W**on't | 这次不会有 | 5% |

## 用户故事拆分

### 拆分原则

1. **按用户角色拆分**
   - 管理员 vs 普通用户
   - 新用户 vs 老用户

2. **按数据量拆分**
   - 单条 vs 批量
   - 小数据 vs 大数据

3. **按操作拆分**
   - CRUD 拆分
   - 步骤拆分

4. **按路径拆分**
   - 主流程 vs 支线流程
   - 正常流程 vs 异常流程

### 拆分示例

**原始故事：**
```markdown
作为 用户，我想管理我的订单，以便跟踪物流
```

**拆分后：**
```markdown
作为 用户，我想查看订单列表
作为 用户，我想查看订单详情
作为 用户，我想追踪物流信息
作为 用户，我想确认收货
```
